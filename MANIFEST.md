# ScanHead Core Architecture Manifest

## 📜 Введение
Этот документ описывает архитектурные принципы, стандарты кодирования и запреты проекта **ScanHead Core**. 
Нарушение принципов манифеста считается критической ошибкой и требует обязательного ревью кода.

---

## 1. 📐 Строгая Многослойность (Layered Architecture)

Проект делится на чёткие изолированные слои. Взаимодействие возможно **только сверху вниз**.

| Слой | Директория | Ответственность | Запреты |
|------|------------|-----------------|---------|
| **UI (Presentation)** | `gui/` | Отрисовка, ввод пользователя, привязка событий. | Никакой бизнес-логики, расчетов, прямых запросов к БД/файлам. |
| **Services (Application)** | `gui/services/` | Бизнес-правила, поиск, генерация штрихкодов, управление состоянием. | Не знают о конкретных виджетах Tkinter. Работают только с интерфейсами. |
| **Domain/Data** | `libs/domain_models/`, `data/` | Чистые модели данных (DTO), интерфейсы репозиториев, контракты. | Не содержат логики приложения или зависимостей от фреймворков. |
| **Infrastructure** | `config/`, `data/mocks/`, адаптеры | Реализация интерфейсов: работа с ФС, БД, принтером, настройки среды. | Не импортируются напрямую в UI. Подключаются через DI. |

**Правило потока данных:**
`UI` → вызывает `Service` → использует `Interface` → реализует `Infrastructure` → возвращает `Domain Model` → обновляет `UI`.

## 2. 🤝 Закон Деметры (Law of Demeter)

**"Разговаривай только с друзьями".** Модуль знает только о тех объектах, которые получает напрямую в качестве аргументов или через зависимости.

### ✅ Разрешено:
- Виджет вызывает метод сервиса: `self._search_service.search_async(query)`.
- Сервис возвращает список моделей: `List[Product]`.
- Связь между компонентами через **Callbacks**, **Events** или **DI-контейнер**.

### ❌ Запрещено:
- Виджет лезет внутрь другого виджета: `details_frame._entry_widget.delete(0, END)` (нарушение инкапсуляции).
- Виджет знает, как сервис ищет данные (путь к файлу, SQL-запрос).
- Сервис манипулирует элементами GUI напрямую.

## 3. 🔌 Инверсия Зависимостей (DIP)

Верхние слои (UI, Services) зависят от **абстракций (Интерфейсов)**, а не от конкретных реализаций.

### Реализация:
1. **Интерфейсы** объявляются в слое Domain (`libs/interfaces/`).
2. **Конкретные реализации** находятся в Infrastructure или Services.
3. **Внедрение** происходит в точке сборки (`main.py`) через `DIContainer`.

```python
# Пример правильного внедрения
class SearchTab:
    def __init__(self, parent, container: DIContainer):
        self._search_service = container.get(ISearchService)
        # UI не знает, Mock это или реальный API
```

Если реализация меняется (например, переход с CSV на PostgreSQL), UI **не требует переписывания**.

## 4. 🧼 Принципы Чистого Кода

### SRP (Single Responsibility Principle)
Один класс — одна ответственность.
- `SearchBar`: управляет только полем ввода и debounce-таймером.
- `SuggestionList`: управляет только отображением выпадающего списка.
- `ProductDetails`: только отрисовка карточки товара.

### DRY (Don't Repeat Yourself)
Повторяющаяся логика (валидация, форматирование, обновление шрифтов) выносится в утилиты или базовые классы.

### KISS (Keep It Simple, Stupid)
Простое решение предпочтительнее "умного".
- Если стандартный виджет ломает UX — пишем свой, но без излишней абстракции.
- Избегаем преждевременной оптимизации.

### Fail Fast
- Проверки `winfo_exists()` перед обращением к виджетам.
- Валидация входных данных в сервисах.
- Ошибки логируются, приложение не падает silently.

## 5. 🔄 Реактивность и Потокобезопасность

### UI пассивен, Сервисы активны
- **UI** слушает события и реагирует на изменения состояния.
- **Сервисы** выполняют тяжелые операции и уведомляют о результате через Callbacks.

### Синхронизация потоков (Tkinter)
Tkinter **не потокобезопасен**.
- ✅ **Разрешено:** Обновление GUI из фонового потока строго через `root.after()` или `queue`.
- ❌ **Запрещено:** Прямой вызов методов виджетов (`label.config()`) из `threading.Thread`.

```python
# Правильный паттерн
def worker():
    result = heavy_task()
    root.after(0, lambda: update_ui(result))
```

## 6. 🧪 Тестируемость и Изоляция

- **Бизнес-логика отделена от Tkinter.** Её можно запускать в headless-режиме (без окна).
- **Моки (Mocks):** В папке `data/mocks/` хранятся заглушки для БД, принтеров и сканеров.
- **Переключение режимов:** Через конфиг (`config/settings.json`). Код приложения не меняется.
- **"Сухой прогон":** Сервисы тестируются отдельно путем подмены реальных зависимостей на фейковые.

## 7. 🚫 Архитектурные Табу (Нарушение = Ревью)

Следующие практики категорически запрещены и подлежат немедленному исправлению:

| Нарушение | Описание |
|-----------|----------|
| ❌ **Бизнес-логика в GUI** | Парсинг строк, расчеты, запросы к файлам/БД внутри файлов `gui/*.py`. |
| ❌ **Прямые связи виджетов** | Обращение к внутренностям соседнего виджета (`other_widget._private_var`). |
| ❌ **Хардкод** | Пути (`C:/...`), токены, строки подключения прямо в коде. Только через конфиг/env. |
| ❌ **GUI из потока** | Прямое изменение состояния UI из background-threads. |
| ❌ **Зависимости в конструкторе** | Создание объектов `Service()` внутри `__init__` виджета. Только DI. |
| ❌ **Глобальные переменные** | Использование `global app_state`. Состояние хранится в сервисах или контейнере. |

## 8. 🗂 Структура Проекта

```text
scanhead-core/
├── libs/
│   ├── domain_models/      # Чистые DTO (Product, Settings)
│   ├── interfaces/         # Абстракции (ISearchService, IPrinter)
│   └── utils/              # Утилиты (logger, validators)
├── gui/
│   ├── widgets/            # Переиспользуемые компоненты
│   ├── tabs/               # Экраны приложения
│   └── services/           # Логика приложения (SearchService, InventoryService)
├── infrastructure/         # Реализации интерфейсов (DB, Files, Hardware)
├── data/
│   ├── mocks/              # Тестовые данные и заглушки
│   └── storage/            # Файлы БД (gitignored)
├── config/                 # Настройки окружения
├── main.py                 # Точка входа, сборка DI-контейнера
└── MANIFEST.md             # Этот документ
```

## 9. 🤝 Процесс Разработки

1. **Ветка:** Все изменения вносятся в feature-ветки от `gui-framework-dev`.
2. **Ревью:** Перед мержем проверяется соответствие Манифесту.
3. **Тесты:** Любая новая фича должна покрываться unit-тестами (для сервисов) или ручным сценарием.
4. **Документация:** Изменения в архитектуре отражаются в этом файле.

> **Помни:** Хороший код — это скучный код. Предсказуемый, читаемый и следующий правилам.
